# 企业智能客服知识库方案

> 基于 AI Agent + 多数据源联动的智能客服系统设计方案
>
> 版本：v1.0 | 日期：2026-01-19

---

## 一、项目背景

### 1.1 业务现状

| 数据资产 | 规模 | 特点 |
|----------|------|------|
| 合作医院 | 10万+ | 持续新增，信息频繁更新 |
| 顾客对话 | 百万级 | 实时产生，包含大量业务经验 |
| 销售话术 | 积累中 | 优秀话术需沉淀复用 |

### 1.2 目标场景

1. **客服AI自动托管对话** - 类似韩国客服AI、美团商户端
2. **内部客服辅助工具** - 帮助客服快速找到合适医院和话术
3. **模拟真人对话** - 有温度、能促单的智能回复

### 1.3 核心诉求

- 能根据顾客画像智能推荐医院
- 能联动后台数据（派单量、成交量、距离等）
- 能匹配合适的话术模板
- 回复要拟人化，促进成交

## 二、场景分析

### 2.1 为什么简单 RAG 不够？

| 维度 | 复杂点 | RAG局限 |
|------|--------|---------|
| **数据规模** | 10万+医院，百万对话 | 向量检索效率下降 |
| **数据动态性** | 医院信息频繁更新 | 向量库更新成本高 |
| **多源联动** | 需同时查医院+顾客+话术 | RAG只支持单一知识库 |
| **结构化查询** | 需要排序、筛选、聚合 | RAG无法处理SQL类需求 |
| **业务逻辑** | 需结合顾客画像做推荐 | RAG缺乏推理能力 |

### 2.2 解决方案：Agent + 多数据源

```
┌─────────────────────────────────────────────────────────────────┐
│                         用户问题                                │
│       "北京口腔医院哪些好" / "给我推荐XX顾客适合的医院和话术"    │
└─────────────────────────────────────────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────────┐
│                      AI Agent（大脑）                           │
│                    理解意图，决定调哪些数据                      │
└─────────────────────────────────────────────────────────────────┘
                                │
            ┌───────────────────┼───────────────────┐

┌───────────────────┐ ┌───────────────────┐ ┌───────────────────┐
│      医院库       │ │      顾客库       │ │      话术库       │
│   (结构化数据)    │ │   (结构化数据)    │ │    (向量数据)     │
├───────────────────┤ ├───────────────────┤ ├───────────────────┤
```

### 3.2 场景A：客服助手（内部工具）

#### 典型问法1：顾客推荐

**输入**："给我推荐XX顾客适合的医院和话术"

**处理流程**：

```
Step 1: 提取顾客ID → 查顾客库
┌─────────────────────────────────────┐
│ 顾客：张三                          │
│ 意向地区：北京朝阳                   │
│ 意向项目：种植牙                     │
│ 预算：1-2万                         │
│ 跟进记录：问过价格，犹豫中           │
└─────────────────────────────────────┘

Step 2: 根据画像 → 查医院库
┌─────────────────────────────────────┐
│ 筛选条件：                          │
│ • 地区 = 朝阳区                     │
│ • 项目包含种植牙                     │
│ • 价格区间匹配                       │
│ • 排序：成交量 DESC                  │
└─────────────────────────────────────┘

Step 3: 根据顾客状态 → 查话术库
┌─────────────────────────────────────┐
│ 场景标签：犹豫型 + 价格敏感          │
│ 匹配话术：打消顾虑 + 性价比话术      │
└─────────────────────────────────────┘
```

**输出示例**：

```
【推荐医院】
1. XX口腔(朝阳店) - 成交128单 - 距离2.3km - 种植牙均价1.2万
2. YY齿科 - 成交96单 - 距离3.1km - 种植牙均价1.5万

【推荐话术】
"张先生，我帮您对比了下，XX口腔这个月种植牙做了128台，
 价格在1.2万左右，离您也近。要不先去免费面诊看看？
 不满意咱再换，反正不花钱~"
```

## 五、数据库设计

### 5.1 医院库（MySQL）

```sql
-- 医院基础表
CREATE TABLE hospitals (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(100) NOT NULL COMMENT '医院名称',
    city VARCHAR(50) NOT NULL COMMENT '城市',
    district VARCHAR(50) NOT NULL COMMENT '区域',
    address VARCHAR(200) COMMENT '详细地址',
    latitude DECIMAL(10,7) COMMENT '纬度',
    longitude DECIMAL(10,7) COMMENT '经度',

    -- 业务属性
    projects JSON COMMENT '支持项目 ["种植牙","矫正","美白"]',
    price_range JSON COMMENT '价格区间 {"种植牙":"8000-15000"}',

    -- 运营数据（定时更新）
    dispatch_count_month INT DEFAULT 0 COMMENT '本月派单量',
    deal_count_month INT DEFAULT 0 COMMENT '本月成交量',
    conversion_rate DECIMAL(5,2) DEFAULT 0 COMMENT '转化率',
    avg_rating DECIMAL(3,2) DEFAULT 0 COMMENT '平均评分',

    -- 状态
    status TINYINT DEFAULT 1 COMMENT '状态 1正常 0下架',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    -- 索引
    INDEX idx_city_district (city, district),
    INDEX idx_deal_count (deal_count_month DESC),
    INDEX idx_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='医院信息表';

-- 医院项目明细表（支持复杂查询）
CREATE TABLE hospital_projects (
    id INT PRIMARY KEY AUTO_INCREMENT,
    hospital_id INT NOT NULL,
    project_name VARCHAR(50) NOT NULL COMMENT '项目名称',
    price_min INT COMMENT '最低价',
    price_max INT COMMENT '最高价',
    deal_count INT DEFAULT 0 COMMENT '成交量',

    INDEX idx_hospital (hospital_id),
    INDEX idx_project (project_name)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='医院项目表';
```

## 六、拟人化回复策略

### 6.1 设计原则

| 维度 | 策略 |
|------|------|
| **称呼** | 亲、您、X先生/女士（根据已知信息）|
| **语气** | 温和友好，像朋友聊天 |
| **节奏** | 短句为主，适当使用语气词（呢、呀、哦）|
| **数据表达** | 口语化（"做了150多台"而非"成交量156"）|
| **引导** | 每次回复都有下一步引导 |

### 6.2 生成模板


## 十、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| **数据质量差** | 推荐不准 | 数据清洗、定期校验 |

| **隐私泄露** | 合规风险 | 数据脱敏、权限控制 |

### B. 常见问题FAQ

**Q1: 为什么不直接用ChatGPT API？**
> 数据安全考虑，医院和顾客数据属于敏感信息，需要私有化部署。

**Q2: Ollama能支撑多大并发？**
> 单机Qwen2.5-7B约支持5-10并发，生产环境建议多实例部署。

**Q3: 话术库多久更新一次？**
> 建议每周更新优秀话术，每月做效果分析淘汰低效话术。
